离线应用和客户端缓存

Dec 13 2017

HTML新增的离线储存和缓存相关的内容

离线检测

为了检测设备是否在线，HTML5定义navigator.onLine属性，是个布尔值，浏览器有差别；当网络发生变化时，会触发两个事件，online和offline

应用缓存

所谓应用缓存即application cache是专门为开发离线web应用设定，APPcache从浏览器缓存中分出一块缓存区域用来保存数据，使用一个描述文件manifest file，列出所需要的下载和缓存的资源，比如<html manifest="sample.appache"></html>

此api的核心为applicationCache对象，有status属性，表示应用缓存的状态，状态改变触发相应的事件

状态

  • 0（UNCACHED） : 无缓存， 即没有与页面相关的应用缓存
  • 1（IDLE） : 闲置，即应用缓存未得到更新
  • 2（CHECKING） : 检查中，即正在下载描述文件并检查更新
  • 3（DOWNLOADING） : 下载中，即应用缓存正在下载描述文件中指定的资源
  • 4（UPDATEREADY） : 更新完成，所有资源都已下载完毕

事件

checking checking事件是整个事件序列中的第一个，表示浏览器在首次下载缓存，或者在检查缓存是否有更新 noupdate, downloading, obsolete, error
noupdate noupdate事件是整个事件序列的最后一个，用来表示缓存没有更新 无
downloading 浏览器正在首次下载menifest中列出的离线资源，或者浏览器发现缓存有更新，并且正在下载该更新 progress, error, cached, updateready
progress 浏览器正在下载menifest中的离线资源，其中progressEvent对象中total表示需要下载的总资源的数量，loaded表示已经处理过的资源数量 progress, error, cached, updateready
cached 浏览器已经成功下载了menifest中的离线资源，应用已经被缓存 无
updateready 缓存有更新并且更新已经下载完成，这是可以使用ApplicationCache.swapCache()来将应用切换到最新缓存 无
obsolete 应用已经有离线缓存了，但menifest文件无法加载成功，则删除现有离线缓存 无
error 发生错误的情况比较多：可能是首次视图缓存离线文件时，发现menifest文件返回404；也可能是下载缓存过程中发生了fatal错误，比如超出了最大可用存储空间的限制，发现了header为Cache-Control: no-store的资源；还有可能是缓存在更新过程中发现menifest文件发生了变化 无

manifest

HTML5的离线存储是基于一个新建的.appcache文件的，通过这个文件上的解析清单离线存储资源，这些资源就会像cookie一样被存储了下来。之后当网络在处于离线状态下时，浏览器会通过被离线存储的数据进行页面展示。

在开始之前要先了解下manifest（即.appcache文件），上面的解析清单要怎么写。

manifest 文件是简单的文本文件，它告知浏览器被缓存的内容（以及不缓存的内容）。
manifest 文件可分为三个部分：

  • CACHE MANIFEST - 在此标题下列出的文件将在首次下载后进行缓存
  • NETWORK - 在此标题下列出的文件需要与服务器的连接，且不会被缓存
  • FALLBACK - 在此标题下列出的文件规定当页面无法访问时的回退页面（比如 404 页面）

在线的情况下,用户代理每次访问页面，都会去读一次manifest.如果发现其改变, 则重新加载全部清单中的资源。

缺点

我们知道每次使用离线应用时，在有网络连接的情况下，浏览器都会逐字节的检查menifest文件是否有更新，而当menifest文件有更新时，就会重新下载menifest文件中列出的所有资源，资源下载成功后会触发updateready事件。这时离线应用本身并不会立即更新，而会在下次访问时才更新，这就是我们所说的二次更新。

  • 超出大小限制的问题:首次下载缓存时超出大小，所有资源都不会缓存，而是请求网络，应用功能正常。
    更新资源后超出大小，缓存不会更新，应用无法更新。
  • 如果有修改的资源文件，必须通过修改manifest文件来属性被缓存的文件列表，只要manifest文件修改，那么所有缓存文件都会被重新拉取一次。更新是全局性的，无法单独更新某个无法单独更新某个文件。
  • 对于链接参数变化是敏感的，任何一个参数修改都会被更新缓存（重复婚车含参页面）index.html和index.html?renew=1会被认为是不同的文件，分别缓存。

数据存储

直接在客户端储存数据，用户的信息

cookie存储

最初用于存储会话信息，可以用来储存一些用户信息，是一对键值对，经过URL编码

特点

  • 绑定在特定的域名之下，确保储存在cookie的信息只能让批准的接受者访问
  • cookie的数量有限，大小有限4k，超过则清除旧的
  • http请求头总会带着存储信息，不安全

构成

名称 唯一确定的名称，不分大小写，经过URL编码

值 存储在cookie中的字符串，经过URL编码

域 cookie对于哪个域是有效的，所有向该域发送的请求都会包含cookie信息

路径 对于指定域中的哪个路径，应向服务器发送cookie

失效时间 表示cookie何时应被删除的时间线，是GMT格式

安全标示 布尔值，指定后，cookie只有在ssl链接时才能发送到服务器

只有键对值才会发送给服务器指示，其他事服务器对浏览器的

javascript中的cookie

通过document.cookie来获取，返回的是cookie的字符串

封装后的cookie的创建，读取，删除功能

web存储机制

克服cookie带来的一系列限制

  • 提供cookie之外的存储会话的途径
  • 提供一种存储大量可跨会话的数据机制
  • 最大容量5M API简单易用

存储内容

数据(序列化)、json数据、图片、脚本、样式文件

storage类型

clear() 删除所有值

getItem(name) 根据指定name获得对应值

key(index) 获得index位置处的值的名字

removeItem(name) 删除由name指定的名值儿

setItem(name,value) 为指定的name设置一个对应的值

数据只保存某个会话，即浏览器关闭对象消失，本地运行不可用

主要用于会话的小段数据的存储，跨域则用localStorage对象

利用length和key来迭代sessionStorage中的值

删除sessionStorage某个值

localStorage对象

属性和方法是一样的

使用注意事项

  1. 浏览器的支持情况
  2. 写入数据时，需要异常处理，避免超出容量报错
  3. 避免敏感信息存入localStorage
  4. key的唯一性

使用限制

  • 存储更新策略，过期控制
  • 子域名之间不能共享储存数据
  • 超出存储大小，利用算法
  • server端如何取得

当然以上的限制，也可通过间接的方式跨越

使用场景

  1. 利用本地数据，减少网络传输
  2. 弱网络环境下，高延迟，低带宽，尽量把数据本地化
